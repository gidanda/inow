# inow 拡張実装ルール

- 対象: MVP 実装以降の全開発
- 目的: 将来機能を追加する際に、既存機能との衝突、責務の混線、破壊的変更を避けるための実装ルールを定義する
- 参照先: 本書を、機能追加・リファクタ・データ構造変更・API追加前に必ず確認する

## 1. 最優先原則

### 1-1. 主役を変えない
- `inow` の主役は常に「マップ」であり、スポット単体を主役にしない
- 新機能追加時も、マップ体験を壊す導線を作らない
- スポット関連機能は「マップを豊かにする要素」として扱う

### 1-2. 既存意味を変えない
- 既存の用語、画面名、API 名、DB カラム名の意味を途中で変えない
- 同じ名前に別の責務を持たせない
- 意味を変えたい場合は既存を流用せず、新しい名前を作る

### 1-3. 追加は基本的に拡張で行う
- 既存の挙動を上書きするのではなく、新しい条件、フィールド、画面、API を追加して拡張する
- 既存仕様を壊す変更は原則禁止とする

## 2. 用語と責務の固定ルール

### 2-1. 用語定義
- `Map` 画面: ホームの地図 UI
- `マップ`: ユーザーが作るコンテンツ単位
- `スポット`: マップに属する地点情報
- `保存マップ`: 他人のマップを保存したもの
- `スポット保存`: 他人のスポットを自分のマップへ複製追加すること

### 2-2. 禁止事項
- `Map` という名前を「画面」と「APIカテゴリ」と「DB種別」で別意味に使い回さない
- `save` という 1 語だけで、マップ保存とスポット複製保存を曖昧に扱わない
- デフォルトマップを通常マップと同じ編集自由度で扱わない

## 3. 機能追加時の境界ルール

### 3-1. 地図基盤と業務機能を混ぜない
- Google Maps Platform は地図表示、Place 検索、位置関連補助に限定する
- `inow` 独自のマップ、スポット、保存、フォロー、ブロック、権限は自前の業務ロジックとして実装する
- Google 由来のデータをそのまま `inow` の業務データとみなさない

### 3-2. 外部データは元データと自前データを分ける
- Google Place 由来スポットでも、`inow` 内では自前スポットとして保存する
- 外部 ID は参照用に保持してよいが、主キーや業務判定の中心にしない
- 外部 API の停止や仕様変更で自サービスの内部整合が壊れないようにする

### 3-3. デフォルトマップは特別扱いを明示する
- `お気に入り` `行きたい` は通常マップとは別ルールを持つ
- デフォルトマップ固有の挙動は `is_default` と `default_type` で明示的に分岐する
- 見た目が似ていても通常マップと同一ロジック前提で実装しない

## 4. API 変更ルール

### 4-1. 既存レスポンスの互換性を壊さない
- 既存 API の必須フィールド削除は禁止
- 既存フィールドの型変更は禁止
- 既存フィールドの意味変更は禁止
- 新しい情報は追加フィールドとして後方互換で足す

### 4-2. 既存エンドポイントの責務を変えない
- 取得 API を別用途へ転用しない
- 既存 API に新責務を過剰に詰め込まない
- 用途が異なる場合は新エンドポイントを追加する

### 4-3. 追加パラメータは任意から始める
- 将来追加する query/path/body パラメータは、原則として既存クライアントを壊さない任意項目で追加する
- 必須化が必要な場合は新 API バージョンまたは新エンドポイントで対応する

### 4-4. 振る舞い分岐は明示する
- `type`, `source_type`, `visibility` などの分岐条件は enum で明示する
- 暗黙の条件分岐を増やさない

## 5. DB 変更ルール

### 5-1. 既存カラムの意味を変えない
- 既存カラムの用途変更は禁止
- 既存 nullable/non-nullable の意味を安易に反転しない
- 意味追加が必要なら新カラムで表現する

### 5-2. 追加は nullable または安全な初期値で入れる
- 新カラム追加時は既存データに影響しない形で入れる
- マイグレーション時に既存行が壊れないようにする

### 5-3. 論理的な種別は enum かフラグで明示する
- 手動スポットと Google Place スポットの違いは `source_type` で持つ
- デフォルトマップか否かは `is_default` と `default_type` で持つ
- 画面やコード側の推測に依存しない

### 5-4. 削除ルールを固定する
- デフォルトマップは削除不可
- 他機能追加時もこのルールを破らない
- 将来的に削除仕様を変える場合は、特別な業務ルールとして明文化してから変更する

## 6. UI/画面追加ルール

### 6-1. 主導線を増やしすぎない
- 下部ナビゲーションの 4 画面構成を安易に崩さない
- 新機能はまず既存画面配下のサブ画面として収められないか検討する
- 独立タブ追加は、既存導線で収まらない場合のみ検討する

### 6-2. 追加行為の文脈を守る
- 作成・追加・編集は `Map`, `Profile`, `User Map` の文脈内で行う
- 追加専用の独立導線を増やしすぎない

### 6-3. 画面責務をまたがせない
- `Profile` は自分の管理
- `User Profile` は他人の公開情報閲覧
- `Saved Maps` は保存マップとデフォルトマップ管理
- `Spot Detail` はスポット詳細
- 役割が異なるものを 1 画面に混ぜない

## 7. 検索追加ルール

### 7-1. 検索の主役は維持する
- `Search` の主軸は常に `Map` 検索とする
- `Spot` と `User` 検索は補助タブとして追加する
- 検索機能追加時も、初期表示がスポット中心にならないようにする

### 7-2. 検索条件は additive に増やす
- 新しいフィルタ条件は既存条件に追加する
- 既存の `タグ`, `エリア`, `距離`, `カテゴリ` の意味を変えない
- 並び順やおすすめ条件は別パラメータで増やす

## 8. 保存機能追加ルール

### 8-1. 2種類の保存を混ぜない
- マップ保存とスポット保存は UI も API も DB も分けて扱う
- 1 つの保存ボタンに複数意味を持たせない

### 8-2. スポット保存は複製である
- スポット保存は「自分のマップへ新規スポットとして複製追加」として扱う
- 元スポット更新への追従はしない
- 将来追従機能を足す場合は別機能として設計する

## 9. 権限追加ルール

### 9-1. 公開範囲の責務はマップ単位
- 公開 / 非公開はマップに対してのみ持つ
- スポット単位の公開制御を MVP に後付けしない
- 将来スポット単位権限を入れる場合は、別仕様として明確に切り出す

### 9-2. 他人データ編集は禁止を維持する
- 他人のマップやスポットを編集可能にする場合は、共同編集などの新しい権限モデルを追加して対応する
- 既存の作成者判定ロジックを流用して曖昧に許可しない

### 9-3. ブロック機能は一覧・検索・プロフィールに波及する
- ブロック機能追加時は `プロフィール表示`, `検索結果`, `フォロー導線`, `マップ閲覧` のすべてに適用範囲を確認する
- 一部画面だけでブロックを効かせる実装は禁止

## 10. 将来機能別の追加指針

### 10-1. コメント機能
- コメントを再導入する場合は `map_comments` を独立テーブルとして追加する
- `Spot` コメントとは混ぜない
- コメント対象が増える場合は polymorphic に逃げず、対象ごとに明示的に設計する

### 10-2. 通知機能
- 通知は既存業務イベントの派生として扱う
- 既存 API のレスポンスに通知都合の状態を直接埋め込みすぎない

### 10-3. 共同編集
- 共同編集は `maps.user_id` の意味を変えず、別の編集権限テーブルで拡張する
- 「所有者」と「編集可能者」は分離する

### 10-4. 限定公開
- `visibility` の意味を上書きせず、`private` に別意味を持たせない
- 限定公開を入れる場合は `visibility = restricted` 等の新値追加で表現する

### 10-5. おすすめ機能
- おすすめは既存検索結果を置き換えず、別セクション・別並び順として追加する
- おすすめロジックを通常検索の結果定義に混ぜない

## 11. 実装前チェックリスト

- この変更はマップ主役の原則を壊していないか
- 既存用語の意味を変えていないか
- 既存 API のレスポンス互換性を壊していないか
- 既存 DB カラムの意味変更で済ませようとしていないか
- マップ保存とスポット保存を混同していないか
- デフォルトマップ固有ルールを壊していないか
- `Profile` と `User Profile` の責務を混ぜていないか
- Google Maps の責務と `inow` の業務責務を混ぜていないか
- ブロックや公開範囲の影響を全画面で確認したか
- 新機能を既存画面に無理に押し込んで責務過多にしていないか

## 12. 運用ルール

- 機能追加前に、本書と `docs/inow_specification.md` を確認する
- 仕様変更が本書のルールに抵触する場合は、実装前に先に文書を更新して合意を取る
- 実装後に本書へ例外ルールを追記する運用は禁止する
- 例外を認める場合は「なぜ既存ルールでは対応できないか」を明文化する
